## Summary

This phase is a pure UI refactor with zero backend or data-flow changes. The goal is to move two existing Qt labels (`iconStatus` and `labelSystemStatusInfo`) from their current parent frame (`systemStatusFrame` inside `ControlPanelController`) into the always-visible `notificationFrame` owned by `MainController`. The freed vertical space (243px) is absorbed by expanding `logViewerFrame` and its child `log_text` widget.

The only non-trivial decision is how `ControlPanelController` retains `self.iconStatus` and `self.labelSystemStatusInfo` as working attributes after they are physically parented to `notificationFrame` (a widget owned by `MainController`). The solution is to create the labels in `MainController._setup_ui()` as children of `notificationFrame`, then pass them as constructor arguments to `ControlPanelController` (or assign them as attributes immediately after construction). This is a simple direct-reference handoff — no signals, no shared state object.

`_update_status_icon` currently hard-codes `71, 71` as the scale target (line ~2235-2238). After the move, the icon lives in a 35x35 geometry, so the pixmap scale target must also be updated to `35, 35`. All three call sites (`~1982`, `~2152`, `~2240`) use `self.labelSystemStatusInfo` and `self.iconStatus` and require no changes other than the attributes being re-pointed to the new labels.

**Primary recommendation:** Create both labels in `MainController._setup_ui()` as children of `notificationFrame`, pass them into `ControlPanelController.__init__` as keyword args `icon_status` and `label_system_status_info`, then assign `self.iconStatus = icon_status` and `self.labelSystemStatusInfo = label_system_status_info` so all existing update code continues working unmodified.

## Architecture Patterns

### Recommended Project Structure

No new files. Changes are contained to two existing files:
- `src/gui/controllers/main_controller.py` — add two QLabel widgets to `notificationFrame`; pass them to `ControlPanelController`
- `src/gui/controllers/control_panel_controller.py` — accept refs, assign as `self.iconStatus`/`self.labelSystemStatusInfo`, remove `systemStatusFrame` block, resize `logViewerFrame` and `log_text`

### Pattern 1: Direct Reference Pass (constructor parameter)

**What:** `MainController` creates the status labels inside `notificationFrame`, then passes them as constructor arguments to `ControlPanelController`. The child controller stores them as `self.iconStatus` and `self.labelSystemStatusInfo`.

**When to use:** When a parent widget owns a child that a page-level controller must update. Avoids coupling through globals, signals, or parent traversal.

**Example:**
```python
# In MainController._setup_ui(), after notificationFrame is created:
self.iconStatus = QLabel(self.notificationFrame)
self.iconStatus.setGeometry(560, 13, 35, 35)
self.iconStatus.setStyleSheet("QLabel { background-color: transparent; }")
self.iconStatus.setAlignment(Qt.AlignCenter)
self.iconStatus.setScaledContents(True)

self.labelSystemStatusInfo = QLabel("Bağlantı Kontrol Ediliyor...", self.notificationFrame)
self.labelSystemStatusInfo.setGeometry(615, 13, 300, 34)  # width TBD by Claude
self.labelSystemStatusInfo.setStyleSheet("""
    QLabel {
        background-color: transparent;
        color: #F4F6FC;
        font-family: 'Plus Jakarta Sans';
        font-weight: 400;
        font-size: 20px;
    }
""")
self.labelSystemStatusInfo.setAlignment(Qt.AlignLeft | Qt.AlignVCenter)

# Then in the ControlPanelController construction call:
self.control_panel_page = ControlPanelController(
    self.control_manager,
    self.data_pipeline,
    parent=self.stackedWidget,
    event_loop=self.event_loop,
    icon_status=self.iconStatus,
    label_system_status_info=self.labelSystemStatusInfo
)
```

```python
# In ControlPanelController.__init__():
def __init__(
    self,
    control_manager=None,
    data_pipeline=None,
    parent=None,
    event_loop=None,
    icon_status=None,
    label_system_status_info=None
):
    ...
    # After super().__init__(parent):
    self.iconStatus = icon_status        # now points to notificationFrame child
    self.labelSystemStatusInfo = label_system_status_info
```

**All existing update calls (`self.iconStatus.setPixmap(...)`, `self.labelSystemStatusInfo.setText(...)`) work unchanged.**

### Pattern 2: Pixmap Scale Update

The `_update_status_icon` method hard-codes `71, 71` as scale dimensions. After the icon is 35x35, update the pixmap scale to match:

```python
# Current (line ~2235):
scaled_pixmap = pixmap.scaled(71, 71, Qt.KeepAspectRatio, Qt.SmoothTransformation)

# After change:
scaled_pixmap = pixmap.scaled(35, 35, Qt.KeepAspectRatio, Qt.SmoothTransformation)
```

### Pattern 3: Log Viewer Expansion

`logViewerFrame` currently ends at y=822 (362 + 460). After removing `systemStatusFrame` (which was at y=823), extend `logViewerFrame` to height 703 so it ends at y=1065, nearly filling the full column.

`log_text` currently: `setGeometry(30, 70, 261, 360)`. With the new frame height of 703, the available interior height is 703 - 70 (top offset) - ~30 (bottom margin) = 603px.

```python
# logViewerFrame: height changes from 460 → 703
self.logViewerFrame.setGeometry(1176, 362, 321, 703)

# log_text: height changes from 360 → 603 (fills new frame)
self.log_text.setGeometry(30, 70, 261, 603)
```

### Anti-Patterns to Avoid

- **Accessing labels via parent traversal (`self.parent().iconStatus`):** Fragile coupling, breaks if hierarchy changes. Use direct reference instead.
- **Using Qt signals to proxy updates:** The existing update methods are already in the GUI thread (called from timer slots). No threading concern exists here.
- **Creating the labels in `ControlPanelController` and re-parenting them:** Qt's `setParent()` call moves a widget to a new parent but also changes its position to (0,0) in the new parent, requiring explicit `setGeometry()` anyway. Creating in `MainController` is cleaner.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Z-order of labels in notificationFrame | Custom z-order management | `raise_()` is already called on `notificationFrame` — children are ordered by creation | QFrame handles child z-order naturally |
| Font size for status text in compact bar | Dynamic font sizing | Fixed font size (20px or smaller) with sufficient label width | Qt clips text at label boundary; just size the label wide enough |

**Key insight:** This is a geometry-and-parenting change only. Qt handles all rendering and event delivery automatically once parent/geometry is set correctly.

## Common Pitfalls

### Pitfall 1: Forgetting to Initialize `self.iconStatus = None` Before `_setup_ui`

**What goes wrong:** The `__init__` body of `ControlPanelController` (lines 530-587) initializes attributes before calling `_setup_ui()`. If `icon_status=None` is passed (e.g., in unit tests that don't construct a full UI), `self.iconStatus` will be `None`. The existing `add_log` method already guards with `if not self.log_text: return` — the same pattern must be applied wherever `self.iconStatus` and `self.labelSystemStatusInfo` are used.

**Why it happens:** Tests or partial initialization paths may not pass live widgets.

**How to avoid:** After accepting `icon_status` and `label_system_status_info` in `__init__`, assign them directly (even if `None`). The existing update methods at ~1982, ~2152, ~2240 already use `self.labelSystemStatusInfo.setText(...)` without guarding — these are safe because the caller always passes real widgets in production. No extra guard needed unless tests call these methods with `None` widgets.

**Warning signs:** `AttributeError: 'NoneType' object has no attribute 'setText'` in tests.

### Pitfall 2: `systemStatusFrame` Removal Leaves Orphan Code

**What goes wrong:** After removing the frame and its three child widgets from `_setup_ui`, there may be other references to `self.systemStatusFrame` elsewhere (e.g., timer stop methods, `stop_timers`, or debug prints).

**Why it happens:** A 2355-line file can have non-obvious references.

**How to avoid:** After editing, grep the file for `systemStatusFrame`, `labelSystemStatus` (the title), and `iconStatus` / `labelSystemStatusInfo` to confirm no stale references remain outside the constructor.

**Warning signs:** `AttributeError: 'ControlPanelController' object has no attribute 'systemStatusFrame'` at runtime.

### Pitfall 3: notificationFrame Height vs. Label Vertical Centering

**What goes wrong:** `notificationFrame` is 60px tall. `iconStatus` at y=13 with height 35 = bottom at 48px (12px margin). `labelSystemStatusInfo` at y=18 with height 34 = bottom at 52px. Both fit inside 60px frame height.

**Why it happens:** Tight frame means off-by-a-few pixel placements cause clipping.

**How to avoid:** Verify coordinates: frame height 60, icon at (560, 13, 35, 35) — bottom=48, ok. Label at (615, 13, W, 34) — bottom=47, ok. Use 13 as the y for both to vertically center within the 60px frame.

**Warning signs:** Icon or text appears clipped at bottom edge of notification bar.

### Pitfall 4: Label Width Too Narrow for Longest Status Text

**What goes wrong:** The longest status messages are multi-line in the old design (e.g., "Kesilecek malzeme\nkonumlandırılıyor."). In the notification bar, word wrap should be OFF and text should be a single line. The label must be wide enough for the longest single-line form.

**Why it happens:** Old `labelSystemStatusInfo` used `setWordWrap(True)` and `AlignCenter` in a 241px-wide frame. In the notification bar the label is left-aligned with no wrap.

**How to avoid:**
- Set `setWordWrap(False)` on the new label
- Use `AlignLeft | AlignVCenter` alignment
- Provide width of at least 350px (longest visible text is ~30 chars at 20px font = ~360px)
- Available space in notificationFrame between x=650 and labelTime at x=1348: ~698px — plenty of room

## Open Questions

1. **Exact `labelSystemStatusInfo` width in notification bar**
   - What we know: Space between x=650 and `labelTime` at x=1348 is ~698px. Longest status text is around 30 characters at 20px font.
   - What's unclear: Whether the font renders wider than estimated at different DPI.
   - Recommendation: Use width=400px as a safe value. If text is clipped in practice, widen. Alternatively use `setMinimumWidth` and let Qt auto-size — but fixed geometry is the established pattern.

2. **`log_text` exact height for 703px frame**
   - What we know: Frame top=362, height=703 → bottom=1065. Label title at y=26-60. Text widget at y=70.
   - What's unclear: Desired bottom margin.
   - Recommendation: Use `log_text.setGeometry(30, 70, 261, 603)` — gives a 30px bottom margin (703 - 70 - 603 = 30), matching the original top-to-text gap visually.

3. **Font size for `labelSystemStatusInfo` in compact bar**
   - What we know: Current font is 26px, but bar height is only 60px (leaving 34px for the label). 26px might render too tall for 34px height.
   - Recommendation: Reduce to 20px to match `labelDate`/`labelTime` style and fit cleanly.
