## Summary

This phase makes two small, self-contained changes to the `NumpadDialog` class in `src/gui/numpad.py` and the two call-sites in `src/gui/controllers/control_panel_controller.py`.

**Change 1 — Close button:** A `QPushButton` (or `QToolButton`) is added to `Ui_Dialog.setupUi()` at absolute position (633, 11) with a 57x57 pixel icon loaded from `src/gui/images/close.png`. Clicking it calls `QDialog.reject()` so the dialog closes without accepting. The call-sites already branch on `result == QDialog.Accepted`, so no call-site changes are needed for the close behavior.

**Change 2 — Pre-fill:** `NumpadDialog` needs to accept an optional `initial_value` string. The constructor should set `self.value` to that string and call `update_label()`. A `_prefilled` flag causes the first digit press to clear `self.value` before appending — standard "replace on first keystroke" behavior. The two call-sites in `control_panel_controller.py` already compute `initial_str` (lines 1605 and 1667) but never pass it; they just need to pass it to the constructor.

**Primary recommendation:** Make all changes in `numpad.py` (add close button to `Ui_Dialog`, add `initial_value` parameter and `_prefilled` flag to `NumpadDialog`). Touch `control_panel_controller.py` only to pass `initial_str` into the constructor at both call-sites.

### Pattern 2: Pre-fill with first-keystroke replacement

```python
class NumpadDialog(QDialog):
    def __init__(self, parent=None, initial_value=""):
        super().__init__(parent)
        self.ui = Ui_Dialog()
        self.ui.setupUi(self)
        self.value = initial_value          # pre-fill
        self._prefilled = bool(initial_value)  # flag: clear on first digit
        self.Accepted = QDialog.Accepted
        self.setup_connections()
        self.update_label()
        # ... rest unchanged

    def add_digit(self, digit):
        """Add a digit to the value."""
        if self._prefilled:
            self.value = ""             # clear pre-filled value on first keystroke
            self._prefilled = False
        if len(self.value) < 10:
            self.value += digit
            self.update_label()
```

Call-site change (two locations, identical pattern):
```python
# Before (lines 1603-1608 and 1665-1670 in control_panel_controller.py)
dialog = NumpadDialog(self)
# Set initial value AFTER construction to avoid showEvent reset
initial_str = str(int(initial_value)) if initial_value > 0 else ""
result = dialog.exec()

# After
initial_str = str(int(initial_value)) if initial_value > 0 else ""
dialog = NumpadDialog(self, initial_value=initial_str)
result = dialog.exec()
```

The stale comment "Set initial value AFTER construction" should be removed — it no longer applies.

### Dead Code Already Present

Lines 1604-1605 and 1666-1667 in `control_panel_controller.py` already compute `initial_str` but never use it. This is dead code left from a previous partial implementation. The fix is simply to pass the value to the constructor and remove the now-stale comments.

### Anti-Patterns to Avoid

- **Setting value after `exec()`:** `showEvent` runs before the user interacts. Setting `self.value` before `exec()` (i.e., in `__init__`) is correct. The existing stale comment warns about `showEvent` reset — that was relevant when value was set after construction; with `__init__` parameter it is a non-issue.
- **Calling `self.reject()` from `Ui_Dialog`:** UI setup class should not know about dialog behavior. Connect `closeButton.clicked` in `NumpadDialog.setup_connections()`, not in `Ui_Dialog`.
- **Modifying `update_label()` for pre-fill:** The existing `update_label()` already handles `self.value` correctly (shows `"0"` when empty). No change needed.

## Common Pitfalls

### Pitfall 1: `showEvent` resets `self.value`

**What goes wrong:** If the close icon image file is missing, the button still appears (empty). The `os.path.exists` guard means no crash — just no icon. The `close.png` file is already in `src/gui/images/` (confirmed: untracked, present on disk).

**How to avoid:** Use the same `os.path.exists` guard already used for backspace/enter icons.

### Pitfall 2: `_prefilled` flag not cleared on `backspace()`

**What goes wrong:** If user presses backspace before a digit (unusual but possible), the pre-filled value should behave normally — backspace removes the last character of the pre-filled string, not trigger a clear.

**How to avoid:** Only clear `_prefilled` in `add_digit()`. Backspace operates on `self.value` normally regardless of `_prefilled` state. This is correct behavior: backspace on a pre-filled value trims it; first digit press replaces it entirely.

### Pitfall 3: `initial_str` computed before dialog construction

**What goes wrong:** Moving `initial_str = ...` above `NumpadDialog(self, ...)` changes execution order slightly. In the existing code, `initial_str` was computed after construction (dead code). The fix moves it to before construction — no behavioral difference, just ordering.

**How to avoid:** Non-issue. Both lines are in the same synchronous method body.

### Pitfall 4: Icon size vs button size

**What goes wrong:** If icon size is set larger than button size, icon is clipped.

**How to avoid:** Set `setIconSize(QSize(57, 57))` to match the button geometry (57x57). The image is already 57x57 per the spec.

## Open Questions

1. **Close button style — transparent vs gradient?**
   - What we know: All existing buttons use the gradient style. Close is a dismiss action, not a value input.
   - What's unclear: User preference for visual consistency vs. distinction.
   - Recommendation: Use transparent/borderless style (within Claude's discretion). Close button should not look like a numpad digit button.

2. **Backspace behavior on pre-filled value**
   - What we know: Spec says "replace on first keystroke." Backspace before any keystroke is an edge case.
   - What's unclear: Should backspace on a pre-filled value trim it (like editing) or clear it (like digits do)?
   - Recommendation: Let backspace trim (normal behavior) — it is the least surprising. Digits replace on first press. This is the more ergonomic choice and avoids a two-step clear.
