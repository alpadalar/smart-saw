## Critical Issues

### CR-01: Plain-text credentials committed to source control

**File:** `config/config.yaml:249-271`
**Issue:** The ThingsBoard device access token (`vtpo7p1jo8mnzln1kdhz`) is hardcoded at three locations in the committed config file (lines 249, 255, 271). The MQTT broker and HTTP host IPs (lines 253, 269) are also committed. Anyone with read access to the repository can impersonate this device or send arbitrary telemetry to the production IoT backend. ## Warnings

### WR-01: Double-inversion logic error in `_run_ldc_inference` produces correct result by accident

**File:** `src/services/camera/ldc_worker.py:267-271`
**Issue:** The per-map post-processing loop already applies `cv2.bitwise_not(m)` (line 260). After the loop, the fused output is inverted a second time with `cv2.bitwise_not(edge_gray)` (line 269), binarized, then inverted *back* (line 271). The net effect on the final image is a no-op compared to a single inversion, meaning the double-inversion cancels out. The algorithm still works because the binarization threshold at 180 (applied to the already-inverted image) is what matters — but the code is misleading and fragile. A reviewer or future maintainer may "fix" one of the inversions and silently break the output.

```python
# Lines 267-271 — current (double-invert then invert-back):
edge_gray = fused                       # fused is already bitwise_not'd
inv = cv2.bitwise_not(edge_gray)        # re-inverts: back to original polarity
_, binary = cv2.threshold(inv, 180, 255, cv2.THRESH_BINARY)
edge_gray = cv2.bitwise_not(binary)     # inverts binary result
```

**Fix:** Clarify intent with a comment explaining that two inversions are deliberate (they implement a "find dark edges on light background" threshold), or collapse them:
```python
# Threshold on the original-polarity fused map (edges are dark, background light)
# We want edge pixels in the output to be white, so we invert after threshold.
_, binary = cv2.threshold(fused, 75, 255, cv2.THRESH_BINARY_INV)
edge_gray = binary
```
If the exact threshold value must stay at 180, keep the current chain but add a multi-line comment explaining each step's purpose and that the double-inversion is intentional.

### WR-02: `calculate_saw_health` returns `100.0` on any exception, masking real errors

**File:** `src/services/camera/health_calculator.py:77-79`
**Issue:** The bare `except Exception` at line 77 catches all exceptions (including `TypeError`, `ZeroDivisionError`, etc.) and silently returns `100.0` — "assume healthy". If, for example, `wear_percentage` is accidentally passed as a string by a caller, the function logs the error but tells the caller the saw is perfectly healthy. Downstream decisions (e.g., alerting operators) will be wrong. The fallback value `100.0` is the worst possible safe default for a safety-monitoring system; a failure-to-compute state should produce an explicit sentinel (e.g., `None`) or re-raise.

**Fix:**
```python
def calculate_saw_health(self, ...) -> float | None:
    try:
        ...
        return saw_health
    except Exception as e:
        logger.error("Saw health calculation error: %s", e, exc_info=True)
        return None  # Let caller decide how to handle unknown health
```
Update callers to handle `None` explicitly (e.g., skip publishing or publish a sentinel value like `-1.0`).

### WR-03: Unchecked empty `preds_uint8` list before indexing at `[-1]`

**File:** `src/services/camera/ldc_worker.py:265`
**Issue:** `preds_uint8[-1]` is accessed immediately after the loop over `maps`. If `maps` is an empty iterable (which is possible if the model returns an empty list), `preds_uint8` will be empty and this line raises `IndexError`, crashing the worker thread with no error recovery. The `run()` method has no outer try/except, so the thread exits silently.

```python
# Line 265
fused = preds_uint8[-1]  # IndexError if maps was empty
```

**Fix:**
```python
if not preds_uint8:
    logger.error("LDC model returned no output maps — skipping cycle")
    self._stop_event.wait(self._interval)
    continue

fused = preds_uint8[-1]
```

### WR-04: `time.sleep(0.15)` in `test_stop_event_exits_loop` is timing-dependent

**File:** `tests/test_ldc_worker.py:362`
**Issue:** The test sleeps for 0.15 s to let the worker enter its loop before calling `stop()`. The `interval_seconds` in the test config is 0.1 s, so the intent is to let at least one iteration run. On a heavily loaded CI system, 0.15 s may not be enough time for the thread to start and reach `_stop_event.wait()`. If it hasn't reached the wait yet when `stop()` is called, the thread will exit on the very next `is_set()` check (which is correct behavior) — but if the thread hasn't even started, `join(5.0)` will wait until the model load path runs. The test could flake on slow machines.

**Fix:** Replace the fixed sleep with a condition-based wait. Since `frame is None` causes an immediate `continue` back to `_stop_event.wait()`, the worker will call `_stop_event.wait()` immediately. Use `worker._stop_event.wait` patching as the other tests do, or check `worker.is_alive()` in a short poll loop with a generous timeout rather than an unconditional sleep:
```python
# Wait for the thread to actually start before stopping
deadline = time.monotonic() + 2.0
while not worker.is_alive() and time.monotonic() < deadline:
    time.sleep(0.01)
worker.stop()
worker.join(timeout=5.0)
```

### IN-01: Duplicate `gui:` top-level key in config.yaml

**File:** `config/config.yaml:12-16` and `config/config.yaml:498-509`
**Issue:** The `gui:` key appears twice at the top level. YAML parsers typically use the last occurrence, silently discarding the first. The first block (lines 12-16) sets `window_width: 1200` and `window_height: 800` but does NOT set `fullscreen` or the update intervals. The second block (lines 498-509) sets `fullscreen: true` and update intervals but drops `window_width` and `window_height`. Whichever block wins depends on the YAML loader; any code reading `gui.window_width` from the merged config will get `None` (key missing) rather than `1200`.

**Fix:** Merge both `gui:` blocks into a single block containing all keys.

### IN-02: `config.yaml` — `camera.wear.roi_y_center_n` is read by config but never used in `_compute_wear`

**File:** `config/config.yaml:554`, `src/services/camera/ldc_worker.py:52`
**Issue:** `roi_y_center_n` is stored in `self._roi_y_center_n` (constructor line 52) and present in all test configs (e.g., `test_ldc_worker.py:36`), but `_compute_wear` never references `self._roi_y_center_n`. The ROI Y coordinates are derived solely from `top_line_y` and `bottom_line_y`. If it is meant to derive `top_line_y`/`bottom_line_y` dynamically from normalized coordinates, implement that calculation.

### IN-03: Broad exception silently swallowed in `calculate_broken_percentage`

**File:** `src/services/camera/health_calculator.py:38-40`
**Issue:** `calculate_broken_percentage` wraps its entire body in `try/except Exception` and returns `0.0` on any error. The calculation is pure arithmetic with an explicit guard for `detected_teeth <= 0`; there is no realistic exception path. The broad catch hides potential programming errors (e.g., receiving a non-numeric type) and returns a misleading `0.0` (zero broken teeth) rather than surfacing the problem.

**Fix:** Remove the try/except entirely. The guard clause already handles the only edge case (`teeth <= 0`). Let unexpected type errors propagate so callers detect them immediately:
```python
def calculate_broken_percentage(self, detected_teeth: int, detected_broken: int) -> float:
    """Return broken-tooth percentage (0-100). Returns 0.0 if teeth <= 0."""
    if detected_teeth <= 0:
        return 0.0
    broken_percentage = (detected_broken / detected_teeth) * 100.0
    return max(0.0, min(broken_percentage, 100.0))
```
